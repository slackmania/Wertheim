# WERTHEIM — Техническая документация

## Обзор

**WERTHEIM** — это автономное приложение для шифрования в виде одного HTML-файла, работающее в браузере. Портировано из оригинального Pascal/Delphi через TypeScript в JavaScript. Цель: обеспечить полную совместимость с форматом `.wtx` при современном интерфейсе, не требующем установки.

---

## Конвейер шифрования

### Кодирование (`hintabe()`)

```
plaintext (строка)
    │
    ▼
UTF-8 encode → Uint8Array
    │
    ▼
RC4 + хеш пароля SHA-1
    │
    ▼
latin1 строка (byte→char, 0..255)
    │
    ▼
encode64() — внутренний Base64
    │
    ▼
bekod() — подстановка Hinta (2 прохода) → encode64() — внешний Base64
    │
    ▼
ciphertext (строка)
```

### Декодирование (`hintaki()`)

```
ciphertext (строка)
    │
    ▼
kikod() — decode64() → обратная подстановка Hinta (2 прохода) → decode64()
    │
    ▼
latin1 строка
    │
    ▼
charCodeAt(i) & 0xFF → Uint8Array
    │
    ▼
RC4 + хеш пароля SHA-1 (расшифровка = повторное шифрование)
    │
    ▼
UTF-8 decode
    │
    ▼
plaintext (строка)
```

## Компоненты подробно

### 1. SHA-1 хеш — Web Crypto API

```javascript
async function sha1(str) {
    const buf = new TextEncoder().encode(str);
    const hash = await crypto.subtle.digest("SHA-1", buf);
    return new Uint8Array(hash);
}
```

Используется встроенная аппаратно-ускоренная реализация SHA-1 браузера. Генерирует 20-байтовый хеш из пароля, необходимый для генерации ключа RC4.

### 2. Шифрование RC4

Нативная JavaScript-реализация, следующая алгоритму из Pascal-файла `titokrutin.pas`. KSA (Key Scheduling Algorithm) и PRNG (Pseudo-Random Number Generator) — стандартные для RC4. Расшифровка идентична шифрованию (свойство XOR).

### 3. Byte-safe конвертация

Критически важный компонент. Выход RC4 — последовательность байтов (0–255). Для обработки в Base64 байты необходимо преобразовать в символы:

**Правильный подход:**
```javascript
String.fromCharCode(...enc)  // byte → char (0–255 → U+0000–U+00FF)
```

**Ошибочный подход (в предыдущей версии):**
```javascript
new TextDecoder("latin1").decode(enc)  // 0x80–0x9F → Windows-1252 многобайтовые
```

`TextDecoder("latin1")` преобразует диапазон 0x80–0x9F в символы Windows-1252 (например, 0x80 → € = U+20AC), что ломает кодировку Base64, так как кодовые точки полученных символов > 255. `String.fromCharCode()` отображает каждый байт 1:1 без изменений.

### 4. Кодирование Base64 (`encode64` / `decode64`)

Нестандартный Base64 — совместимый с Pascal-оригиналом. Алфавит ABC0 (`"ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/"`) совпадает с Pascal-версией. Кодирование преобразует 3 байта в 4 символа, заполнение символом `=`.

### 5. Подстановка Hinta

**Концепция:** Для каждого символа пароля существует строка таблицы перестановок (`tarr`). Строка `i` содержит перестановку всех символов ABC0. Каждый символ текста заменяется символом из соответствующей строки и столбца.

**Реализация:**

```javascript
function genhinta(pwd) {
    const pwdenc = encode64(pwd);            // → Base64
    const trimmed = pwdenc.replace(/=+$/, ""); // обрезка заполнения
    const kod = new Array(MAX_PWD_LEN);       // индексы строк
    for (let t = 0; t < pwdhossz && t < MAX_PWD_LEN; t++) {
        kod[t] = pwdenc.charCodeAt(t) % TARR_ROWS;
        hintar[t] = tarr[kod[t]][col];
    }
}
```

**Важное замечание:** Индекс `col` здесь — `pwd.charCodeAt(t)`, где `pwd` — это `encode64(password)`. Кодовые точки символов Base64 находятся в диапазоне 43–122, но массивы `tarr[row]` содержат только 65 индексов (0–64). Таким образом, `hintar[t]` для большинства паролей будет `undefined` (в Pascal-оригинале это неопределённое поведение, которое работает идентично). Это **не влияет на корректность round-trip**, так как ошибочный индекс симметрично проявляется при кодировании и декодировании.

**Роль Hinta:** Двухпроходная подстановка добавляет дополнительное перемешивание к выходу Base64. Хотя из-за вышеуказанной ошибки индекса столбца текущая реализация с большинством паролей практически не изменяет текст Base64 (подстановка заменяет символы пустой строкой), это симметрично, поэтому декодирование работает корректно.

### 6. Таблица `tarr`

Таблица перестановок 64×65, где каждая строка содержит случайную перестановку всех символов ABC0. Сгенерирована Pascal-программой, вшита в исходный код (`tarr_data.ts` → `wertheim.html`).

## Анализ безопасности

### Слабые стороны

1. **Ошибка индекса столбца Hinta:** В `genhinta()` индекс `pwd.charCodeAt(t)` выходит за пределы массива `tarr[row]` для большинства паролей. Это фактически отключает подстановку Hinta. Ошибка симметрична, поэтому совместимость не нарушается, но стойкость шифрования снижается.

2. **Известные уязвимости RC4:** RC4 не считается современным стандартом. Смещения в первых байтах и известные статистические атаки.

3. **Отсутствие аутентификации:** Целостность сообщения ничем не защищена. Зашифрованный текст может быть изменён без обнаружения.

### Сильные стороны

1. **Byte-safe конвертация:** `String.fromCharCode()` обеспечивает отображение 1:1 каждого байта, в отличие от ошибочного поведения `TextDecoder("latin1")`.

2. **Нет внешних зависимостей:** Один файл, нет CDN, нет стороннего кода.

3. **Zero-trust архитектура:** Нет сервера, нет отправки данных, нет телеметрии.
