# WERTHEIM — التوثيق التقني

## نظرة عامة

**WERTHEIM** هو ملف HTML مستقل يعمل بالكامل في المتصفح. وهو منقول من تطبيق Pascal/Delphi الأصلي إلى TypeScript/JavaScript. الهدف هو التوافق الكامل مع تنسيق ملف `.wtx` مع توفير واجهة مستخدم حديثة لا تتطلب تثبيتًا.

## خط أنابيب التشفير

### الترميز (`hintabe()`)

```
plaintext (string)
    │
    ▼
UTF-8 encode → Uint8Array
    │
    ▼
RC4 + SHA-1 (هاش كلمة المرور)
    │
    ▼
latin1 string (byte→char, 0..255)
    │
    ▼
encode64() — Base64 داخلي
    │
    ▼
bekod() — استبدال hinta (دورتان) → encode64() — Base64 خارجي
    │
    ▼
ciphertext (string)
```

### فك الترميز (`hintaki()`)

```
ciphertext (string)
    │
    ▼
kikod() — decode64() → عكس hinta (دورتان) → decode64()
    │
    ▼
latin1 string
    │
    ▼
charCodeAt(i) & 0xFF → Uint8Array
    │
    ▼
RC4 + SHA-1 (هاش كلمة المرور) (فك التشفير = إعادة تشفير)
    │
    ▼
UTF-8 decode
    │
    ▼
plaintext (string)
```

## تفاصيل المكونات

### 1. هاش SHA-1 — Web Crypto API

```javascript
async function sha1(str) {
    const buf = new TextEncoder().encode(str);
    const hash = await crypto.subtle.digest("SHA-1", buf);
    return new Uint8Array(hash);
}
```

يستخدم تطبيق SHA-1 المدمج في المتصفح والمُسارَع عبر الأجهزة. ينتج هاشًا بطول 20 بايت من كلمة المرور، يُستخدم لجدولة مفتاح RC4.

### 2. تشفير RC4

تطبيق JavaScript أصلي يتبع الخوارزمية من `titokrutin.pas`. يشمل خوارزمية جدولة المفاتيح القياسية (KSA) ومولد الأرقام العشوائية الزائفة (PRNG). فك التشفير مماثل للتشفير (خاصية XOR).

### 3. تحويل البايت الآمن

المكون الأكثر حساسية. مخرجات RC4 هي سلسلة من البايتات (0–255). للتعامل معها كـ Base64، يجب تحويل البايتات إلى أحرف:

**النهج الصحيح:**
```javascript
String.fromCharCode(...enc)  // byte → char (0–255 → U+0000–U+00FF)
```

**النهج الخاطئ (في الإصدار السابق):**
```javascript
new TextDecoder("latin1").decode(enc)  // 0x80–0x9F → Windows-1252 multi-byte
```

`TextDecoder("latin1")` يحول البايتات 0x80–0x9F إلى أحرف Windows-1252 (مثل 0x80 → € = U+20AC)، مما يكسر ترميز Base64 لأن نقاط الرمز الناتجة > 255. `String.fromCharCode()` يحول كل بايت 1:1 دون تحويل.

### 4. ترميز Base64 (`encode64` / `decode64`)

Base64 غير قياسي — متوافق مع Pascal الأصلي. يستخدم أبجدية من 64 حرفًا `ABC0` (`"ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/"`)، مطابقًا لإصدار Pascal. يشفر 3 بايتات إلى 4 أحرف، مع حشو بـ `=`.

### 5. استبدال Hinta

**المفهوم:** كل حرف من كلمة المرور يقابل صفًا في جدول التبديل (`tarr`). الصف `i` يحتوي على تبديل لجميع أحرف ABC0. يتم استبدال كل حرف في النص بالحرف الموجود في الصف والعمود المناسبين.

**التنفيذ:**

```javascript
function genhinta(pwd) {
    const pwdenc = encode64(pwd);            // → Base64
    const trimmed = pwdenc.replace(/=+$/, ""); // إزالة الحشو
    const kod = new Array(MAX_PWD_LEN);       // مؤشرات الصفوف
    for (let t = 0; t < pwdhossz && t < MAX_PWD_LEN; t++) {
        kod[t] = pwdenc.charCodeAt(t) % TARR_ROWS;
        hintar[t] = tarr[kod[t]][col];
    }
}
```

**ملاحظة هامة:** يستخدم مؤشر `col` القيمة `pwd.charCodeAt(t)` حيث `pwd` هي `encode64(password)`. نقاط رمز أحرف Base64 تتراوح بين 43–122، لكن مصفوفات `tarr[row]` تحتوي فقط على 65 مؤشرًا (0–64). لذلك `hintar[t]` ستكون `undefined` لمعظم كلمات المرور (محاكاة السلوك غير المحدد في Pascal الأصلي). هذا **لا يؤثر على صحة الرحلة ذهابًا وإيابًا** لأن المؤشر الخاطئ متماثل في الترميز وفك الترميز.

**دور hinta:** الاستبدال على دورتين يضيف مزجًا إضافيًا على مخرجات Base64. على الرغم من أن خطأ مؤشر العمود يجعله عديم الفائدة عمليًا لمعظم كلمات المرور (الاستبدال بسلاسل فارغة)، فإن التماثل يعني أن فك التشفير لا يزال يعمل بشكل صحيح.

### 6. جدول `tarr`

جدول تبديل بأبعاد 64×65 حيث يحتوي كل صف على تبديل عشوائي لجميع أحرف ABC0. تم إنشاؤه بواسطة برنامج Pascal الأصلي ومضمن في الكود المصدري (`tarr_data.ts` → `wertheim.html`).

## تحليل الأمان

### نقاط الضعف

1. **خطأ مؤشر عمود hinta:** تستخدم `genhinta()` القيمة `pwd.charCodeAt(t)` والتي تتجاوز حدود مصفوفة `tarr[row]` لمعظم كلمات المرور. هذا يعطل طبقة استبدال hinta عمليًا. الخطأ متماثل، لذا يتم الحفاظ على التوافق، لكن قوة التشفير تقل.

2. **نقاط ضعف RC4 المعروفة:** RC4 ليس معيارًا حديثًا. لديه تحيزات في وحدات البايت المبكرة للمخرجات وهجمات إحصائية معروفة ضده.

3. **لا يوجد توثيق:** سلامة الرسالة غير محمية. يمكن تعديل النص المشفّر دون اكتشاف.

### نقاط القوة

1. **تحويل بايت أمين:** يضمن `String.fromCharCode()` أن كل بايت يُخطط 1:1، على عكس `TextDecoder("latin1")` الذي يفسد البايتات 0x80–0x9F.

2. **لا تبعيات خارجية:** ملف واحد، لا CDN، لا كود طرف ثالث.

3. **تصميم عدم الثقة:** لا خادم، لا نقل بيانات، لا تليمتري.
