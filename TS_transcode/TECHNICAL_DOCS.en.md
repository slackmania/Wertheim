# WERTHEIM — Technical Documentation

## Overview

**WERTHEIM** is a standalone HTML file that runs entirely in the browser. It is a port of the original Pascal/Delphi application to TypeScript/JavaScript. The goal is full `.wtx` file format compatibility while providing a modern, zero-install user interface.

---

## Encryption Pipeline

### Encoding (`hintabe()`)

```
plaintext (string)
    │
    ▼
UTF-8 encode → Uint8Array
    │
    ▼
RC4 + SHA-1 password hash
    │
    ▼
latin1 string (byte→char, 0..255)
    │
    ▼
encode64() — inner Base64
    │
    ▼
bekod() — hinta substitution (2 rounds) → encode64() — outer Base64
    │
    ▼
ciphertext (string)
```

### Decoding (`hintaki()`)

```
ciphertext (string)
    │
    ▼
kikod() — decode64() → hinta reverse (2 rounds) → decode64()
    │
    ▼
latin1 string
    │
    ▼
charCodeAt(i) & 0xFF → Uint8Array
    │
    ▼
RC4 + SHA-1 password hash (decrypt = re-encrypt)
    │
    ▼
UTF-8 decode
    │
    ▼
plaintext (string)
```

## Component Details

### 1. SHA-1 Hash — Web Crypto API

```javascript
async function sha1(str) {
    const buf = new TextEncoder().encode(str);
    const hash = await crypto.subtle.digest("SHA-1", buf);
    return new Uint8Array(hash);
}
```

Uses the browser's built-in, hardware-accelerated SHA-1 implementation. Produces a 20-byte hash from the password, used for RC4 key scheduling.

### 2. RC4 Encryption

Native JavaScript implementation following the algorithm from `titokrutin.pas`. Includes standard Key Scheduling Algorithm (KSA) and Pseudo-Random Generation Algorithm (PRNG). Decryption is identical to encryption (XOR property).

### 3. Byte-Safe Conversion

The most critical component. RC4 output is a sequence of bytes (0–255). To handle these as Base64, bytes must be converted to characters:

**Correct approach:**
```javascript
String.fromCharCode(...enc)  // byte → char (0–255 → U+0000–U+00FF)
```

**Wrong approach (previous version):**
```javascript
new TextDecoder("latin1").decode(enc)  // 0x80–0x9F → Windows-1252 multi-byte
```

`TextDecoder("latin1")` maps bytes 0x80–0x9F to Windows-1252 characters (e.g., 0x80 → € = U+20AC), breaking Base64 encoding because resulting code points > 255. `String.fromCharCode()` maps every byte 1:1 without transformation.

### 4. Base64 Encoding (`encode64` / `decode64`)

Non-standard Base64 — compatible with the Pascal original. Uses a 64-character alphabet `ABC0` (`"ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/"`), matching the Pascal version. Encodes 3 bytes into 4 characters, padding with `=`.

### 5. Hinta Substitution

**Concept:** Each password character corresponds to a row in a permutation table (`tarr`). Row `i` contains a permutation of all ABC0 characters. Each character in the text is substituted with the character at the corresponding row and column.

**Implementation:**

```javascript
function genhinta(pwd) {
    const pwdenc = encode64(pwd);            // → Base64
    const trimmed = pwdenc.replace(/=+$/, ""); // remove padding
    const kod = new Array(MAX_PWD_LEN);       // row indices
    for (let t = 0; t < pwdhossz && t < MAX_PWD_LEN; t++) {
        kod[t] = pwdenc.charCodeAt(t) % TARR_ROWS;
        hintar[t] = tarr[kod[t]][col];
    }
}
```

**Important note:** The `col` index uses `pwd.charCodeAt(t)` where `pwd` is `encode64(password)`. Base64 characters have code points 43–122, but `tarr[row]` arrays only have 65 indices (0–64). Therefore `hintar[t]` will be `undefined` for most passwords (mirroring the Pascal original's undefined behavior). This **does not affect round-trip correctness** since the wrong index is symmetric in encoding and decoding.

**Role of hinta:** The 2-round substitution adds additional mixing on top of Base64 output. Although the column index bug makes it effectively a no-op for most passwords (substituting with empty strings), the symmetry means decryption still works correctly.

### 6. `tarr` Table

A 64×65 permutation table where each row contains a random permutation of all ABC0 characters. Generated by the original Pascal program and hardcoded into the source (`tarr_data.ts` → `wertheim.html`).

## Security Analysis

### Weaknesses

1. **Hinta column index bug:** `genhinta()` uses `pwd.charCodeAt(t)` which exceeds `tarr[row]` array bounds for most passwords. This effectively disables the hinta substitution layer. The bug is symmetric, so compatibility is preserved, but encryption strength is reduced.

2. **RC4 known weaknesses:** RC4 is not a modern standard. It has biases in early output bytes and known statistical attacks.

3. **No authentication:** Message integrity is not protected. Encrypted text can be modified without detection.

### Strengths

1. **Byte-faithful conversion:** `String.fromCharCode()` ensures every byte maps 1:1, unlike `TextDecoder("latin1")` which corrupts bytes 0x80–0x9F.

2. **No external dependencies:** Single file, no CDN, no third-party code.

3. **Zero-trust design:** No server, no data transmission, no telemetry.
